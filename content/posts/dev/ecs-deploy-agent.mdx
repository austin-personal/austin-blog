# ECS Deploy Agent 사용해서 배포하기

> sub-title: ECS란?

## 사건

- 사내 팀에서 만든 ECS deployer agent가 잘 작동하지 않았다. 배포는 되었지만 계속 서버가 다운되고 재시작 되었다.
- 나름 AWS 자격증이 있는 내가 그냥 지나칠 수 없었다. 자세히 파악해보기로 했다.

## ECS 란?

이미지만 있으면 쉽게 컨테이너를 실행시킬 수 있고 여러 개의 컨테이너를 쉽게 관리할 수 있도록 도와주는 AWS 클라우드 서비스이다.

### 구성요소

1. Cluster: 인프라 자원을 공유하는 네임스페이스이다. "이 안에 있는 Service와 Task는 같은 환경이다"라는 경계선 역할 정도로 생각하면 좋다. 하나의 클러스터 안에 여러 태스크 데피니션을 통해 생성된 서비스들이 운용된다.

- 그래서 보통 환경별로 분리한다 Dev cluster, Stage, Prod
- 네임스페이스, 인프라 자원 설정, 공통 룰 (ecs exec 설정, 모니터링 등등)

2. Task definition: "컨테이너를 어떻게 띄워라"라는 블루프린트 (Json) 이다. 이 이미지가 어떤 인프라 자원을 얼마나 사용할 건지, 네트워크 설정, 환경변수, 로깅, 헬스체크 등을 설정할 수 있다. ~~AWS 관련 설정을 제외하고 Dockerfile과 유사한 설정들과 비슷하다.~~ docker run 옵션들과 비슷하다고 보면 된다.

- Json: 이 이미지가 어떤 인프라 자원을 얼마나 사용할 건지, 네트워크 설정, 환경변수, 로깅, 헬스체크 등을 설정

3. Task: 블루프린트(Task definition)으로 생성된 컨테이너. 이 이미지가 어느 서버에서 어떻게 동작하고 있는지 파악할 수 있다. 마치 도커 데스크탑에서 하나의 띄워진 컨테이너

4. Service: 태스크들을 정해진 규칙에 맞게 운영한다. 운영 = 재시작 로직, 인프라 관리, 스케일링, Rolling Update, roll back, 트래픽 라우팅 설정

- 어느 클러스터에서 실행할지, 태스크 정의 패밀리, 운영 관리 설정

요리와 비유:

0. 도커 이미지: 밀키트 음식 재료 팩
1. Task definition: 조리방법
2. Task: 조리법을 통해 만들어진 음식
3. Service: 만들어진 음식을 음식이 상하지 않게 관리하는 역할 (음식과 음식에 적합한 진열대를 매치)
4. Cluster: 종류별로 나눈 음식 진열대 - 종류별 = 같은 칸에 보관 (같은 인프라 그룹)

비록 짧은 경험이지만 ECS는 다양한 컨테이너를 관리하기 적합하게 느껴졌다.

### 그외

1. ECS Agent

- 기본적으로 ECS는 Fargate이든 아니든 EC2에서 동작한다. ECS Agent는 EC2 위에 도커 데몬에게 어떤 컨테이너를 어떻게 실행해라라고 명령하는 EC2에서 상주하는 매니저이다.
- ECS agent는 EC2에 실행되는 데몬 프로세스이다. 이 데몬 프로세스가 ECS control plane에 long polling을 하며 할 일이 있는지 지속적으로 체크한다.

  > Daemon process란? 원래 부모 프로세스가 종료되어 PID 1(init/systemd)에 입양되어, 다른 서비스 의존 없이 백그라운드에서 계속 실행되는 프로세스
  > 왜 폴링인가? 푸시가 아니라?
  >
  > 1. 푸시하려면 ECS 컨트롤 플레인이 여러 개의 ECS agent(EC2)에 접근 설정을 해야함 subnet, ec2의 IP, ec2의 인바운드 포트 등등
  > 2. 푸시하려면 ECS가 각 EC2 ECS agent(ec2)와 연결을 관리해야함
  > 3. 폴링하면 메모리 대역폭을 점유는 하지만 위와 같은 ECS control plane에 부하가 없음

- ECS 에서 서비스 생성 -> ECS agent가 EC2에서 docker run ... 실행 -> 도커 데몬 해당 명령어 실행
- 헬스체크도 이 친구가 함 -> 헬스체크 실패시 서비스한테 "UNHEALTHY" 해요라고 전달 -> 서비스는 재시작 로직에 따라 재시작 결정함 -> ECS agent가 EC2에서 도커 명령어 실행

2. ECS Replica Service VS Daemon Service

- Service는 태스크 데피니션이 특정 클러스터에서 운용될 수 있도록 설정하는 것이다.
- 서비스에는 두가지 유형이 있음

> 대몬 - 이것이 기본 전략입니다. 스케줄러는 클러스터에 지정된 태스크 배치 제약 조건을 모두 충족하는 각 활성 컨테이너 인스턴스에 한 태스크씩 정확히 배치합니다.
> 복제본 - 스케줄러는 클러스터 전체에 지정된 태스크 수를 배치하고 유지 관리합니다. 기본적으로 서비스 스케줄러는 가용 영역에 태스크 분산합니다. -AWS-

## Claude ECS 배포 에이전트

### 현재 상황

현재 우리팀은 AI 서비스들을 쉽게 배포하도록 Agent를 만들어 배포하고 있다.
API 통신이 필요한 서비스들은 하나의 ALB를 통해 각 서비스들로 트래픽이 전해지도록 설정했다. (아직 규모가 작고 도메인 발급이 귀찮기때문)

### 순서

~~0. 클러스터 생성: 라우팅, 인프라 설정~~

1. 해당 도커 파일을 빌드하고 ECR에 저장한다.
2. 에이전트가 도커 파일을 파악
3. 태스크 데피니션 초안을 작성한다. 환경변수등 시크릿들을 SSM parameter store에 저장하고 태스크 데피니션 초안에 secret 주소를 적는다.
4. 특정 조건에 태스크 데피니션 초안과 공유 클러스터 내용 그리고 도커이미지를 바탕으로 태스크 데피니션을 작성한다.
5. 클러스터에 해당 태스크 데피니션을 바탕으로 서비스를 생성한다. -> 태스크 생성 완료! = 배포 완료!

## 사건의 원인과 해결

### 원인

그래서 왜 안되었나? 원인은 한 팀원이 바꾼 헬스체크 로직때문이었다.

> 태스크 데피니션 작성시 이 태스크가 잘돌고 있는지 확인하는 헬스체크 설정이 있다. 이 헬스체크 커맨드를 바탕으로 ECS agent가 EC2에서 이 컨테이너가 잘 떠있는지 확인한다.

이 Crash loop의 원인은 curl에서 wget으로 헬스체크 로직을 바꾼 이유였다.

1. wget: HTTP/HTTPS/FTP에서 파일을 다운로드하는 CLI 도구

- 이름 유래: World Wide Web + get

2. curl: URL을 통해 데이터를 전송/수신하는 CLI 도구.

나는 파이썬 슬림 이미지를 기반으로 사용하고 있었다. 최소한의 런타임만 남기고 나머지 다 삭제한 이미지이기 때문에 기본적으로 wget이 존재하지 않는다. 그래서 ecs agent가 헬스체크를 하려고 하는데 1을 반환해서 unhealthy라고 판단하고 재시작을 진행한 것이다.

| 이미지 | wget | curl |
|---|---|---|
| python:3.12-slim | X | X |
| python:3.12-alpine | O | X |
| python:3.12 (full) | O | O |
| node:22-slim | X | X |
| node:22-alpine | O | X |
| node:22 (full) | O | O |

- Alpine 계열은 기본적으로 BusyBox 기반
  > Busybox란? [BusyBox(비지박스)는 GPL 라이센스로 개발되고 있는, 400여 개의 리눅스 커맨드라인 명령어들(wget포함)을 모아놓은 단일 실행 파일](https://www.lainyzine.com/ko/article/busybox/)
- full 이미지에는 이것저것 다 들어가 있음

### 해결

- 에이전트에 베이스 이미지 확인하고 wget 사용 명시적으로 넣도록 변경
